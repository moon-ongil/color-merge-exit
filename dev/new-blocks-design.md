# 신규 블록 설계 문서 — 타이머 블록 & 색-순환 블록

> 상태: 제안(설계만). 코드 변경 없음. 작성 2026-07-14.
> 핵심 원칙: **모든 신규 메커닉은 "수(move) 기반"으로 설계한다.** 그래야 기존
> `Solver`(턴제 상태 그래프)로 생성 시 "풀림 보장"을 유지할 수 있다. 실시간(벽시계)
> 타이머로 가면 1000레벨 전수 솔버 검증이라는 이 게임 최대 강점을 잃는다.

관련 코드 앵커:
- 규칙: `Assets/Scripts/Core/Board.cs`(`TryMove`/`MaxSlide`/merge·exit·split), `Block.cs`, `Enums.cs`(`ColorMix`)
- 데이터: `Assets/Scripts/Core/LevelData.cs`(JSON 스키마), `Exit.cs`
- 솔버: `Assets/Scripts/Core/Solver.cs`(`Neighbors`/`IsSolvable`/`Hint`) — 상태 키 = 블록(id,x,y,color,locked) + 도어 인덱스
- 세션/UX: `GameSession.cs`, `Assets/Scripts/Game/GameController.cs`, `BoardView.cs`, `TutorialStore.cs`
- 생성: `dev/mergegen.py`(`solve()`는 실제 해답을 cap 내 찾은 후보만 채택), `dev/gen_extend.py`

## 공통 원칙: 왜 "수 기반"인가
현재 `Solver`의 상태 전이는 `(블록 배치/색, 도어 인덱스) --한 수--> 다음 상태`로 완전히
결정적이다. `IsSolvable`(데드엔드 감지)과 `Hint`, 생성기 `solve()`가 모두 이 그래프를 탐색한다.

- 블록 상태가 **벽시계 시간**에 의존하면(예: "5초마다 색 변경") 같은 배치라도 시각에 따라
  전이가 달라져 그래프가 비결정적이 되고 → 생성 시 풀림 보장을 못 한다.
- **"N수마다"로 정의**하면 전이가 여전히 결정적 → 솔버가 그대로 검증한다.

두 블록 모두 아래 설계는 수 기반이다.

---

## 블록 A — 타이머 블록(Fuse Block): "먼저 빼야 하는 블록"

### 컨셉
스폰 시 `fuse = N`(예: 6). **플레이어가 한 수 둘 때마다 1 감소.** fuse가 0이 되기 전에
이 블록을 보드에서 제거(머지되어 사라지거나 도어로 탈출)해야 한다.

### 규칙(정확히)
- 카운트다운 단위: **플레이어의 유효 수(Moved/Merged/Split/Exited)** 1회당 fuse -= 1.
  (Blocked/Invalid는 소모 안 함 — `MoveResult` 참고)
- fuse 감소 시점: 그 수의 결과가 확정된 직후. 자기 자신이 이번 수로 제거되면 감소 무시(이미 성공).
- **만료(fuse가 0인데 아직 보드에 있음) 처리** — 아래 3안 중 택1:
  - (A안·권장) **즉시 실패**: `GameSession`을 `Lost`로. time-attack과 일관되고 규칙이 단순.
  - (B안) **잠금 벽으로 변함**: 그 자리에서 `locked` + 불변 장애물화. 계속 플레이 가능하나
    "먼저 빼라"는 압박이 약해지고, 남은 퍼즐이 풀리는지 재검증 필요.
  - (C안) **폭발**: 인접 블록 제거/교란. 화려하지만 솔버 모델·생성 난이도 급상승 → 비권장.
- 타이머 블록도 일반 블록처럼 색/모양을 가지며 머지·탈출 규칙은 동일. fuse는 "언제까지"만 추가.

### 데이터 스키마 (`BlockSpawnData` 확장)
```jsonc
{ "id": 5, "color": 0, "x": 3, "y": 2, "cells": [...],
  "fuse": 6 }        // 0 또는 미지정이면 일반 블록
```
`Block`에 `int Fuse`(0=없음) 필드 추가. `LevelData.BuildBoard`에서 전달.

### 솔버 영향 (`Solver`)
- **상태 키에 각 타이머 블록의 남은 fuse 포함** (또는 전역 move-count 포함). 즉 배치가 같아도
  fuse가 다르면 다른 상태.
- `Neighbors`: 각 전이(한 수)에서 살아있는 모든 타이머 블록의 fuse -= 1. fuse가 음수가 되며
  아직 보드에 남은 분기는 **가지치기**(그 상태에서 목표 도달 불가로 취급, A안 기준).
- 영향: 상태 공간이 fuse 차원만큼 늘지만 유한(각 블록 fuse ≤ 초기값). 탐색 비용 증가하나
  현재 랜덤-재시작 솔버로 감당 가능(16블록 레벨도 초 단위 해결 확인됨).
- **데드엔드 감지 자동 연동**: fuse가 촉박해 못 빼는 배치는 `IsSolvable`이 "풀 수 없음"으로
  잡아 기존 경고/중단 흐름을 그대로 탄다.

### 생성 (`mergegen.py`)
- `solve()`는 이미 전체 해답을 찾으므로, **해답 상의 "이 타이머 블록이 몇 번째 수에 제거되는가"**를
  읽어 `fuse`를 그 값 이상으로 살짝 여유 있게 설정하면 항상 만족 가능.
- 난이도 램프: 레벨 id가 오를수록 fuse 여유(slack)를 줄여 압박 강화. 초기 도입 레벨은 slack 크게.

### UX / 비주얼 (`BoardView`)
- 블록 위 **숫자 배지(남은 수)**. fuse ≤ 2면 빨강 펄스 + 사운드로 경고.
- 레벨 자체 제한시간과 혼동 방지: 블록 배지는 "숫자만"(초 아님)으로 표기, 필요시 아이콘(💣/⏱) 구분.

## 블록 B — 색-순환 블록(Chameleon Block): "타이밍 맞춰 머지"

### 컨셉
지정한 **색 사이클**(예: Red→Blue→Yellow→Red…)을 따라 **N수마다 다음 색으로** 자동 변경.
맞는 색일 때 머지/탈출해야 한다. 엉뚱한 색으로 머지하면 원치 않는 혼합색이 생겨
**데드엔드**가 될 수 있다.

### 규칙(정확히)
- `cycle = [색...]`, `cyclePeriod = N`(기본 1 = 매 수마다 1칸 이동). 현재 색 = `cycle[phase]`.
- **phase 진행 시점**: 플레이어의 유효 수마다 카운터 +1, `cyclePeriod`마다 phase = (phase+1) % len.
  (period=1이면 매 수 색 변경; 2면 두 수마다)
- 머지/탈출은 일반 규칙과 동일하되, **판정에 쓰이는 색은 "그 수 시점의 현재 색"**.
- **"잘못 머지 = 못 끝냄"은 별도 실패 처리 불필요**: 이미 있는 데드엔드 감지(`IsSolvable`)가
  "이 배치는 풀 수 없음"을 증명해 기존 흐름으로 처리. 단 **UX상 undo 허용/경고를 강력 권장**
  (한 수 실수로 강제 리트라이는 이탈 유발).

### 데이터 스키마 (`BlockSpawnData` 확장)
```jsonc
{ "id": 7, "x": 2, "y": 4, "cells": [...],
  "cycle": [0, 1, 2],   // CarColor 인덱스들; 있으면 color 필드 대신 cycle[0]로 시작
  "cyclePeriod": 1 }
```
`Block`에 `CarColor[] Cycle`, `int CyclePeriod`, `int CyclePhase` 필드 추가.

### 솔버 영향 (`Solver`) — **가장 깔끔한 부분**
- 이미 상태 키에 **각 블록의 현재 color가 포함**돼 있다. 그래서 색-순환 블록은:
  - `Neighbors`에서 **한 수 둘 때마다 각 색-순환 블록의 color를 사이클 다음 값으로 진행**시키면 끝.
  - 별도의 move-counter를 상태에 넣을 필요 없음(색 자체가 상태의 일부라서 자동 구분됨).
    period>1이면 phase(0..period-1)만 소형 필드로 상태에 추가.
- 즉 **기존 solver 모델에 최소 변경**으로 결정적 검증 가능. 데드엔드도 자동 감지.
- 생성기 `mergegen.solve()`도 동일 방식으로 색을 진행시키면 그대로 "풀림" 검증.

### 생성 (`mergegen.py`)
- 후보에 색-순환 블록을 배치 → `solve()`가 "매 수 색 진행" 규칙으로 실제 해답을 찾은 경우만 채택.
- 난이도: 사이클 길이↑(3→4색), period=1(빠른 변화)일수록 어려움. 초기 도입은 짧은 사이클/period 크게.

### UX / 비주얼 (`BoardView`)
- 블록에 **다음 색 미리보기**(작은 점 인디케이터)와 "N수 후 변경" 힌트. 색 전환은 부드러운 트윈.
- 데드엔드 위험: 잘못 머지 직전 **경고 링/흔들림**, 그리고 **undo 버튼 유지**.

## 난이도 곡선 & 도입(튜토리얼)
- 둘 다 **고급 메커닉** → 후반부 도입 권장. 예: 색-순환은 600레벨~, 타이머는 700레벨~
  (기존 램프처럼 id 함수로 등장 빈도/강도 상승; `mergegen.build()` 램프에 편입).
- 튜토리얼: 기존 `GameController.MaybeShowTutorial` + `TutorialStore` 패턴 재사용
  (예: `tut.chameleon`, `tut.fuse` 최초 등장 레벨에서 1회 코치마크).
- 한 레벨에 신규 2종 동시 등장은 초반엔 피하고, 각각 단독 학습 후 조합.

## 확정 필요한 결정(Open Questions)
1. **타이머 만료 처리**: A(즉시 실패) / B(잠금 벽으로 변함) — 게임 톤에 따라.
2. **색-순환 period**: 1(매 수) vs 2~3(여러 수). 매 수는 어렵고 긴장감↑.
3. **잘못 머지**: 데드엔드 자동감지에 맡기고 undo 권장 vs 즉시 명시적 "실패" 연출.
4. 도입 레벨 구간과 등장 빈도 램프 수치.
5. 아트: 두 블록의 배지/아이콘 디자인(`dev/asset_sheet_prompt.md` 연계).
